binaryview

binaryninja的分析都基于binaryview对象

在3.5.4378版本以前是通过open_view方法获得一个binaryview对象,不过在这个版本被使用load方法获取所替代

load的第一个参数可以是一个文件也可以是一个字节流,不过这里只讨论文件

symbols

bv对象的一个字典成员,键为符号的名称,值为一个成员为符号的列表

一个符号有如下成员

  • address: int
    Symbol address (read-only)
  • auto: bool
    Whether the symbol was auto-defined
  • binding: SymbolBinding
    Symbol binding (read-only)
  • full_name: str
    Symbol full name (read-only)
  • handle
  • name: str
    Symbol name (read-only)
  • namespace: NameSpace
    Symbol namespace (read-only)
  • ordinal: int
    Symbol ordinal (read-only)
  • raw_bytes: bytes
    Bytes of the raw symbol (read-only)
  • raw_name: str
    Symbol raw name (read-only)
  • short_name: str
    Symbol short name (read-only)
  • type: SymbolType
    Symbol type (read-only)

符号的类型有如下

FunctionSymbol Symbol for function that exists in the current binary
ImportAddressSymbol Symbol defined in the Import Address Table
ImportedFunctionSymbol Symbol for a function that is not defined in the current binary
DataSymbol Symbol for data in the current binary
ImportedDataSymbol Symbol for data that is not defined in the current binary
ExternalSymbol Symbols for data and code that reside outside the BinaryView
LibraryFunctionSymbol Symbols for functions identified as belonging to a shared library
SymbolicFunctionSymbol Symbols for functions without a concrete implementation or which have been abstractly represented
LocalLabelSymbol Symbol for a local label in the current binary

vuleak

使用Binary Ninja进行IoT设备漏洞挖掘 | 京东探索研究院信息安全实验室 (jd.com)

之前一直认为写一个漏洞发现工具是一件困难的事情,而本人的写代码水平也一直不咋滴,直到在漏洞挖掘的学习过程中发现了上面这篇文章给了我不小的启示,只要比人工效率更高,那就是进步,所以我在这篇文章的提供的代码基础上自己完善了一下这个工具,并暂且称之为vuleak

binaryninja是一个不错的选择,除了强大的通用api外,其独特的BNIL中间语言,允许我们进行更为精细化的操作

第二个改进是针对mips架构参数判断逻辑修复,在mips架构下进行库函数的调用往往是先将其加载到$t9寄存器,这样get_code_refs代码引用得到的就是la $t9,func这样的指令地址

而不是我们需要的跳转处,这时可能并不一定会先加载参数,所以此处会出现较大纰漏,导致返回参数为undetermined

我修复了这一情况,不过最终结果打印的信息中地址尚未更改,应该不影响orz

2.1

在对另一个程序进行测试时发现无法正常获取参数,经过调试发现binarninja的llil指令并不是按照地址顺序排列的

之前的逻辑是llil索引加1来找到跳转参数,现在由于这个修改为addr+=4,暂时解决这个问题

2.2

一波未平一波又起,上面那个问题的解决方案又带来了一个新问题,即mips是可能有8字节长度的指令的

如果遇到这样的情况那就会导致返回il_index时返回None从而导致错误

所以增加了一个il_index检查

2.3

刚解决上面的问题又遇到一个新问题

这次的问题是之前的判断跳转逻辑仅仅是针对jalr,而实际可能会遇到更多的跳转指令

所以判断跳转指令逻辑改为判断llil指令的operation是否是跳转属性

2.4

在测试时又发现可能会陷入死循环,原因是一些情况下函数地址被加载到$t9寄存器后,不一定会被调用,导致判断跳转指令逻辑陷入死循环

于是增加了两个计数器,当到达一定次数即停止并返回None

3

第三个改动是优化了is_constant的判断,只有在值处于.rodata节时才会被判为constant

不过这还存在着一些问题,例如如果是一个整型常量,那么就会判断失效,不过对于system,strcpy等函数来说,这也并不是一个大问题

于是更改为判断const指向域所在段是否有写权限,暂时解决问题

4

增加了对strcpy,strcat,strcmp,strncmp等函数的check支持,不过只有一些简单的过滤规则
